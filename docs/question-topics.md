# 题目 Topic 体系

## 1. 目标

这份文档只定义题目 topic 的产品层组织方式，不定义题库数据结构、字段设计、分发算法和实现细节。

它要解决的问题是：

1. 系统到底按什么 topic 组织 California M1 笔试内容
2. 哪些 topic 属于关键覆盖范围
3. topic 如何服务练习、复习、模拟考试和 readiness 判断

## 2. 为什么要单独定义 topic

如果没有独立的 topic 体系，题目系统虽然可以出题，但后续很多能力都会失去统一基础：

- 练习无法判断用户现在学到哪一块
- 错题无法稳定归类到知识范围
- 复习无法知道该补基础、补细节还是补混淆点
- mock exam 后的薄弱点总结会变得不稳定
- readiness 无法判断关键范围是否真正覆盖

所以 topic 不是“给题目打标签”这么简单。  
它是整个学习路径的内容骨架。

## 3. Topic 不等于题目，不等于章节

当前产品里，topic 需要和两个概念区分开：

- 它不等于单道题
- 它也不直接等于 handbook 的原始章节目录

原因是：

- 单道题太细，无法承载学习路径和覆盖判断
- 原始章节是知识来源，不一定适合直接做学习组织
- 产品需要围绕“更容易通过考试”组织内容，而不是围绕“文档排版”组织内容

因此当前定义是：

- topic 是用户能被持续学习、持续复习、持续评估的一块稳定知识范围

## 4. Topic 体系的组织原则

### 原则 1：范围要稳定

一个 topic 一旦定义，应该在练习、错题、复习、mock exam、summary 里都能复用。  
不能今天按“标志”，明天按“道路规则细节”，后天又按“易错题型”重新拆一套。

### 原则 2：粒度要适中

topic 不能大到“所有交通规则”，也不能小到“左转时头部检查的某一个动作”。  
它应该足够大，能承载学习路径；也要足够小，能识别薄弱点。

### 原则 3：围绕考试失分组织

topic 的边界不能只考虑知识归类，还要考虑：

- 哪些内容高频失分
- 哪些内容高混淆
- 哪些内容更适合单独拉出来反复补

### 原则 4：topic 之下再按学习层次推进

topic 解决“学什么”。  
学习层次解决“学到多深”。

也就是说：

- 先确定 topic
- 再在 topic 内按基础概念、规则细分、相似规则辨析、场景判断推进

### 原则 5：允许跨 topic 混淆，但不混掉主归属

有些题天然会同时碰到多个知识范围。  
例如 lane splitting 题可能同时涉及速度、空间、风险判断和车辆位置。

当前产品规则是：

- 每道题要有一个主 topic
- 系统可以在后续文档里支持相关 topic 或混淆点关系
- 但当前产品层先不把 topic 体系设计成复杂多主归属模型

## 5. Topic 的分层方式

当前建议分三层理解：

### 第一层：主 topic

这是用户侧能感知、系统侧能稳定追踪的核心知识范围。

### 第二层：子 topic

这是主 topic 内更细的内容块，用于后续支持更精细的复习和覆盖分析。  
MVP 阶段先定义方向，不强制每个子 topic 都展开到参数级别。

### 第三层：学习层次

这不是新的 topic，而是同一 topic 内的学习深度：

- 基础概念
- 规则细分
- 相似规则辨析
- 场景判断

## 6. 当前建议的主 Topic 列表

MVP 阶段建议至少稳定覆盖以下主 topic：

1. 交通标志与信号
2. 标线与车道规则
3. 路权与路口优先规则
4. 速度控制与安全距离
5. 转向、变道与位置选择
6. 超车、跟车与车流互动
7. lane sharing / lane splitting
8. 头盔、防护装备与车辆基础安全
9. 酒精、药物、疲劳与风险驾驶
10. 危险识别与防御性骑行
11. 特殊道路与特殊场景
12. 停车、路边操作与基础法律责任

这套列表的目标不是完美复刻手册目录，而是保证：

- 新用户能从常见规则进入
- 高频混淆点有稳定归属
- 关键 topic 覆盖可以被追踪
- mock exam 后能输出有解释力的薄弱点总结

## 7. 各主 Topic 的当前定义方向

### 1. 交通标志与信号

范围示例：

- 常见监管标志
- 警告标志
- 指示性标志
- 交通信号灯与基本信号含义

它解决的是“看见标志或信号时你先要知道它代表什么”。

### 2. 标线与车道规则

范围示例：

- 黄线、白线、双线等基本含义
- 车道边界与可否跨越
- 专用车道或特殊车道的基本判断

它解决的是“路面标线本身在要求你做什么或不能做什么”。

### 3. 路权与路口优先规则

范围示例：

- 停止、让行、先行顺序
- 路口、并入、转弯时的优先关系
- 行人、自行车、其他车辆相关优先规则

它是典型高价值 topic，因为很多题不只是记规则，而是判断谁先谁后。

### 4. 速度控制与安全距离

范围示例：

- 基本速度观念
- 与路况、能见度、风险相匹配的速度判断
- 跟车距离与缓冲空间

它更多服务“速度不是背数字，而是背后的安全判断”。

### 5. 转向、变道与位置选择

范围示例：

- 左转、右转前后的基本动作
- 观察、打灯、位置选择
- 变道前后的检查与优先判断

这是典型需要从基础动作推进到场景判断的 topic。

### 6. 超车、跟车与车流互动

范围示例：

- 什么时候适合超车
- 什么时候不该超车
- 在车流中如何保持安全互动

它和速度、位置、风险判断相关，但仍应保持独立主 topic。

### 7. lane sharing / lane splitting

范围示例：

- 合法性与边界理解
- 何时更危险
- 与周围车流速度差、空间和位置的关系

这是 California motorcycle 场景里非常值得单独定义的重点 topic。

### 8. 头盔、防护装备与车辆基础安全

范围示例：

- 头盔与眼部保护
- 基础防护装备
- 车辆基本安全检查意识

它既有直接规则题，也有“最佳做法”类题目。

### 9. 酒精、药物、疲劳与风险驾驶

范围示例：

- 酒驾、药物影响
- 疲劳对判断和控制的影响
- 风险行为带来的后果

这类题通常不是机械记忆，而是明确风险判断和禁止边界。

### 10. 危险识别与防御性骑行

范围示例：

- 扫描与观察
- 可见性
- 空间管理
- 预判其他道路使用者行为

它是很多场景题和“最佳行动”题的基础。

### 11. 特殊道路与特殊场景

范围示例：

- 高速路
- 弯道
- 交叉口密集区域
- 湿滑、夜间、恶劣天气等场景

它用于承接那些不能只靠基础规则回答的场景类内容。

### 12. 停车、路边操作与基础法律责任

范围示例：

- 基本停车规则
- 路边起步或临停相关注意点
- 与骑行直接相关的基础法律责任

这个 topic 不一定题量最大，但需要保证基础覆盖完整。

## 8. 关键 Topic 的当前产品结论

并不是所有 topic 对 readiness 的影响都完全相同。  
MVP 阶段应明确存在一组“关键 topic”。

当前建议至少把以下范围视为关键 topic：

- 路权与路口优先规则
- 标线与车道规则
- 转向、变道与位置选择
- 速度控制与安全距离
- lane sharing / lane splitting
- 酒精、药物、疲劳与风险驾驶
- 危险识别与防御性骑行

原因是这些内容更直接影响：

- 高频失分
- 场景判断稳定性
- mock exam 后是否还能维持表现
- readiness 是否可以保守地上调

这不代表其他 topic 不重要。  
而是表示在 readiness 判断里，关键 topic 的未覆盖成本更高。

## 9. Topic 与复习路径的关系

topic 只是知识范围，不直接等于复习路径。  
但复习路径必须建立在 topic 之上。

当前关系应当是：

- 某个 topic 基础不稳时，进入基础补齐路径
- 某个 topic 已有基础但细节不稳时，进入同 topic 深化路径
- 某个 topic 存在高混淆错误时，进入混淆点强化路径
- 多个关键 topic 已基本覆盖后，才更适合进入考前验证路径

所以复习系统输出的不只是“你错了哪道题”，而是“你在哪个 topic 的哪一层出了问题”。

## 10. Topic 与 mock exam 的关系

mock exam 不是按单一 topic 连续出题，但它必须依赖 topic 体系。

至少要支持这些产品目标：

- 覆盖主要 topic，而不是集中在少数范围
- 能识别关键 topic 是否缺失
- 能在考试后输出“最影响通过率的薄弱 topic”

这意味着：

- mock exam 的分数是真实结果
- 但 mock exam 后的总结必须回到 topic 维度

## 11. Topic 与 readiness / 完成度的关系

当前产品已经明确：

- 高分不等于 ready
- 真实成绩和 readiness 不能混为一谈

topic 体系在这里承担的职责是：

- 判断关键 topic 是否已经覆盖
- 判断用户是否只在少数范围表现好
- 判断哪些高风险 topic 还没有补完

因此 readiness 至少要依赖：

- 分数表现
- 关键 topic 覆盖情况
- 关键难点是否完成复习

其中“关键 topic 覆盖情况”必须建立在这份 topic 体系上。

## 12. Topic 与用户可见表达的关系

MVP 阶段建议用户侧也能感知 topic，但不必过度暴露复杂结构。

用户侧更适合看到的是：

- 你最近最弱的 3 个 topic
- 你今天优先该补的 topic
- 你还没覆盖的关键 topic

不建议一开始就把完整复杂 taxonomy 全部直接摊给用户。  
产品应该先让 topic 服务行动，而不是服务浏览。

## 13. 当前不在这份文档里定义的内容

这份文档暂时不定义：

- 每个 topic 的精确题量
- 每个 topic 的子 topic 完整枚举
- topic 覆盖阈值
- readiness 里各 topic 的权重
- mock exam 的 topic 配比
- 题目多 topic 关联的数据模型
- 混淆点关系的存储方式

这些都要等 topic 体系先稳定后，再进入参数层和数据层文档。

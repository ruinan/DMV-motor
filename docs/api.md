# API 设计

> **文档分层说明**
>
> 这份文档（`docs/api.md`）是**产品层 API 边界定义**，描述系统需要暴露哪些能力、服务什么产品动作、有哪些状态校验约束。
>
> 开发实现时请以 [`docs/development/api-contract.md`](development/api-contract.md) 为准，那份文档定义了具体的 HTTP 路径、请求响应字段、错误码、鉴权头和分页格式。
>
> 两份文档的关系：`api.md` 是上游产品规则，`api-contract.md` 是下游开发合同。如果两者有出入，以 `api-contract.md` 为准并反馈修正 `api.md`。

## 1. 目标

这份文档定义产品层 API 边界，回答这些问题：

- 系统需要暴露哪些能力接口
- 每类接口服务什么产品动作
- 哪些接口需要状态校验
- 哪些接口之间存在调用顺序约束

这份文档不定义：

- HTTP 路径命名细节
- 请求响应字段清单
- 鉴权实现细节
- 控制器与服务层拆分
- SDK 设计

## 2. API 设计原则

### 原则 1：按产品动作分组，不按数据库表分组

API 设计应该围绕用户和系统行为，例如：

- 开始练习
- 提交答案
- 获取今日复习包
- 开始 mock exam

而不是围绕表名机械暴露增删改查。

### 原则 2：分数、readiness、访问权限必须走独立判断链路

这三类状态都很关键，不能因为某个接口的便捷更新而相互混淆。

### 原则 3：双语是同一能力的表达，不是重复造两套接口

中英双语不应该做成两组 API。  
语言应该是同一能力下的显示或内容参数。

### 原则 4：mock exam 必须有更强的状态约束

mock exam 和普通练习不同：

- 有配额限制
- 退出即结束
- 结果要形成完整快照

所以相关 API 必须显式校验状态。

## 3. API 模块总览

当前产品至少需要以下 API 模块：

1. Auth / Account
2. Access / Subscription
3. Practice
4. Mistakes
5. Review
6. Mock Exam
7. Summary / Progress / Readiness
8. Reminder
9. Language
10. Memory Export
11. Admin / Content Ops

## 4. Auth / Account API

### 主要职责

- 注册或登录
- 获取当前用户基本状态
- 重置学习进度
- 获取账户级学习概览

### 关键能力

- 创建或恢复用户身份
- 读取当前 access 状态
- 读取当前语言偏好
- 读取当前是否存在未完成练习或复习

### 关键约束

- 重置学习进度不等于删除历史数据
- 登录后应能恢复当前学习主状态

## 5. Access / Subscription API

### 主要职责

- 查询免费或付费访问状态
- 激活 30 天访问期
- 查询剩余 mock exam 次数

### 关键能力

- 返回当前是否处于免费体验期
- 返回是否拥有有效 AccessPass
- 返回当前剩余 mock exam 配额
- 返回是否允许进入完整复习和 mock exam

### 关键约束

- access 状态必须是所有受限能力的统一判断来源
- mock exam 配额不能在多个接口各自维护

## 6. Practice API

### 主要职责

- 开始一轮普通练习
- 拉取下一题
- 提交单题答案
- 中断并恢复练习
- 完成一轮练习

### 关键能力

- 创建 PracticeSession
- 按当前规则返回题目
- 按语言偏好返回对应 QuestionVariant
- 提交后返回对错和简短解释
- 更新错误记录与后续复习输入

### 关键约束

- 免费用户只能访问固定 30 题体验集，可反复做，无消耗计数（不存在"剩余额度"概念）
- 已提交题目要保留结果
- 未提交题目不能计分
- 普通练习允许恢复

## 7. Mistakes API

### 主要职责

- 查看错题列表
- 查看错题详情
- 查看某个 topic 下的持续薄弱点

### 关键能力

- 返回活跃错题
- 返回每道错题对应的 topic 和最近错误状态
- 返回错题来源是普通练习还是 mock exam

### 关键约束

- Mistake API 主要提供回看能力
- 它不直接替代 Review API 的任务组织能力

## 8. Review API

### 主要职责

- 获取今日复习包
- 获取复习任务详情
- 开始或恢复复习
- 提交复习作答
- 完成某项复习任务

### 关键能力

- 返回当前 ReviewPack
- 返回包内 ReviewTask 列表
- 按任务目标组织题目
- 更新任务完成状态
- 在复习结束后返回下一步建议

### 关键约束

- 复习任务必须和错题、薄弱点、topic 覆盖状态联动
- 复习包允许恢复
- 复习任务完成后应触发 progress / readiness 更新

## 9. Mock Exam API

### 主要职责

- 查询是否可开始 mock exam
- 开始一次 mock exam
- 拉取 mock exam 题目
- 提交单题答案或阶段结果
- 交卷
- 主动退出并结束
- 获取 mock exam 结果

### 关键能力

- 校验当前 access pass 和剩余配额
- 创建 MockAttempt
- 锁定本次试卷内容
- 交卷后返回真实分数
- 返回薄弱 topic 总结和下一步复习建议

### 关键约束

- mock exam 不允许像普通练习那样自由恢复
- 用户一旦确认退出，本次 mock exam 必须结束
- 配额消耗必须有明确时机，且全链路一致

### 当前建议的配额消耗时机

产品层建议：

- 成功开始一次 mock exam 时即消耗一次配额

原因是：

- 避免用户反复开启退出来规避限制
- 保持“开始验证”本身就是一次资源消耗

## 10. Summary / Progress / Readiness API

### 主要职责

- 获取学习总结页
- 获取 completion 状态
- 获取 readiness 状态
- 获取当前推荐下一步动作

### 关键能力

- 返回最近学习表现
- 返回关键 topic 覆盖情况
- 返回当前主要薄弱点
- 返回 readiness 是否达到 ready 候选
- 返回未满足的硬门槛

### 关键约束

- 真实分数和 readiness 必须分开展示
- API 不能把高分直接映射成 ready
- readiness 输出必须与参数文档中的硬门槛一致

## 11. Reminder API

### 主要职责

- 查询当前待提醒任务
- 生成提醒任务
- 标记提醒已发送或已处理

### 关键能力

- 根据未完成复习、薄弱点、学习中断、mock exam 时机生成提醒候选
- 根据客户端能力决定可用触达渠道
- 返回提醒优先级

### 关键约束

- 提醒逻辑属于后端统一判断能力
- 提醒不能脱离 readiness 和 review 状态单独存在

## 12. Language API

### 主要职责

- 获取当前语言偏好
- 切换显示语言
- 查询当前可用语言

### 关键能力

- 在不改变学习状态的前提下切换语言
- 让同一道题返回对应语言版本

### 关键约束

- 语言切换不能影响题目归属、作答记录、错题和 readiness
- 不允许因为语言切换生成两套学习历史

## 13. Memory Export API

### 主要职责

- 发起学习记忆导出
- 查询导出任务状态
- 获取导出结果

### 关键能力

- 创建 MemoryExport
- 返回导出是否完成
- 返回是否加密、是否设备绑定

### 关键约束

- 导出的是学习记忆和备考状态，不是开放数据库导出
- 读取权限需要受当前安全策略约束

## 14. Admin / Content Ops API

### 主要职责

- 管理 topic
- 管理 question
- 管理 question 双语版本
- 管理 mock exam 试卷定义

### 关键能力

- 维护主 topic 与关键 topic 标记
- 维护题目主归属和语言版本
- 维护题目是否能进入练习、复习、mock exam

### 关键约束

- 同一道题的中英文版本必须共享同一个学习对象
- 后台内容管理不能破坏用户已有学习记录的稳定归属

## 15. 关键 API 顺序约束

### 练习主链路

1. 登录并恢复账户状态
2. 校验访问权限
3. 开始 PracticeSession
4. 提交 PracticeAttempt
5. 更新错题与复习输入
6. 刷新 summary / readiness

### 复习主链路

1. 获取 ReviewPack
2. 进入某个 ReviewTask
3. 提交复习作答
4. 更新任务状态
5. 刷新 progress / readiness

### mock exam 主链路

1. 校验配额与访问期
2. 开始 MockAttempt
3. 作答
4. 交卷或退出结束
5. 生成结果快照
6. 返回复习建议

## 16. 哪些 API 需要最强状态保护

当前最需要强状态校验的是：

- 开始 mock exam
- mock exam 退出
- mock exam 交卷
- 激活 AccessPass
- 重置学习进度
- 导出学习记忆

这些接口都不应只做简单请求转发。

## 17. 当前不在这份文档里定义的内容

这份文档暂时不定义：

- REST 还是 GraphQL
- 路由命名
- 分页格式
- 错误码体系
- 幂等策略
- 权限中间件实现
- 后台和前台是否拆成独立服务

这些内容要在后续技术架构文档里再定。

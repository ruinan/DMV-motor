# 复习与 Readiness 引擎设计

## 状态

- 当前状态：进行中
- 推进方式：问答式逐步定稿
- 当前问题：错题记录在什么情况下创建，以及什么情况下保持活跃

## 已确认结论

- 题库采用“共享题库 + 个性化投放”的模型
- `Question` 是共享内容对象，用户学习状态挂在共享题目之上
- 个性化主要基于 `MistakeRecord`、review 状态和整体学习状态
- 系统允许基于用户薄弱点生成少量新的针对性题目，但这属于个性化投放增强层，不替代共享题库本体
- `MistakeRecord` 在用户每次答错时创建或更新
- `MistakeRecord` 主粒度先按 `question` 建立
- `MistakeRecord` 同时挂上该题的 `main topic`
- 同一题再次答错时，更新累计错误次数和最近出错时间
- topic 层薄弱点判断属于 review/readiness 引擎的派生逻辑，不作为第二套主记录维护
- `MistakeRecord` 不因答对一次就立刻失活
- 只有当用户完成相关复习，并在后续相关题目中表现出稳定正确后，`MistakeRecord` 才失活
- 如果之后再次答错，则重新激活或继续更新该 `MistakeRecord`
- 当前阶段先定义规则层原则，不先定量化算法
- `ReviewTask` 的生成触发条件包括：
- 普通练习答错
- mock exam 答错
- 关键 topic 覆盖不足
- 活跃 `MistakeRecord` 长时间未处理
- mock 后暴露出明显影响 readiness 的薄弱点
- `ReviewTask` 的优先级主要由以下维度共同决定：
- `risk level`
- `readiness impact`
- `foundation gap`
- `mistake persistence`
- `recency`
- `mock signal`
- `coverage gap`
- `staleness`
- 当前优先级倾向为：
- 高风险且显著影响 readiness 的问题优先
- 基础概念缺口优先于细节辨析
- mock 暴露的问题优先于普通练习中的一般问题
- 反复出错或长期未解决的问题优先
- 普通覆盖不足排在更后面
- `ReviewPack` 是从 `ReviewTask` 池中收敛出来的执行切片，不是活跃任务全集
- 当任务很多时，只取当前容量内最有价值的一批，其余留在任务池等待后续轮次
- 当任务较少或暂时没有活跃任务时，可用关键覆盖补齐、轻量巩固题或 readiness 前验证题补足
- `ReviewPack` 收敛时要遵守这些约束：
- 先补基础，再进入细分辨析
- 避免单一 topic 长时间霸占整包
- 适度加入验证是否真正掌握的内容
- 支持未完成后续做，也允许下一轮生成时部分刷新和替换
- `completion` 先于 `readiness`
- `readiness` 建立在 `completion` 之上，而不是反过来
- `completion` 采用“覆盖完成度 + 初步掌握”的表达
- `completion` 至少拆成两个指标：
- 覆盖完成度
- 掌握情况
- 这意味着 `completion` 不只是“看过多少”，也不要求达到 `readiness` 那样严格的考试状态
- `readiness` 至少需要经过一组硬门槛约束，而不是只看单次高分
- 当前确认的硬门槛候选包括：
- mock exam 真实成绩达到最低门槛
- 关键 topic 覆盖没有明显缺失
- 高风险活跃薄弱点不能过多
- 关键 review task 不能大量未完成
- 最近表现需要具备一定稳定性
- `readiness` 的目标不是“偶尔能考高分”，而是让 P90 最差情况下也仍有较高概率通过正式考试
- 具体阈值与统计口径后续放到参数层定义和调优
- mock exam 结束后，错题需要以更高权重回流到 review
- mock 中的错题先写入题目结果明细，再更新或激活对应 `MistakeRecord`
- 如果某个问题在 mock 中暴露，且之前已经存在 `MistakeRecord`，则说明该薄弱点更顽固，后续 review 强度应进一步提高
- mock 暴露出的高风险问题应直接生成更高优先级的 `ReviewTask`
- 新一轮 `ReviewPack` 应优先吸收本次 mock 暴露的问题
- 如果 mock 表现很好，不是直接清空 review，而是降低易错题或已明显改善问题的优先级，逐步转向验证型和收尾型复习

## 决策记录

- 决策：题库本体采用共享题库，用户看到的投放顺序与局部补强内容可个性化
- 原因：共享题库保证内容稳定、双语一致、错题复用和跨端一致性；个性化投放保证复习针对性
- 决策：允许基于用户错题记录与薄弱点生成少量新的针对性题目
- 原因：用户不应只重复看到原题，系统需要围绕同一薄弱点补充变体题
- 决策：`MistakeRecord` 在每次答错时创建或更新，主粒度为 `question`
- 原因：question 级状态最稳定，topic 级判断更适合作为派生结果而非第二套主状态
- 决策：`MistakeRecord` 不因单次答对立即失活，而是在相关复习完成且后续表现稳定后才失活
- 原因：单次答对不等于真正掌握，需要和“高分不等于 ready”的整体产品原则保持一致
- 决策：`ReviewTask` 不只由单题错误触发，也可由 topic 覆盖不足、长期未处理薄弱点、mock 暴露问题触发
- 原因：review 系统目标是补强学习路径，而不是只做错题回放
- 决策：`ReviewTask` 优先级由风险、readiness 影响、基础缺口、错误持续性、mock 信号等多维度共同决定
- 原因：不是所有错题都同样重要，review 系统应优先处理最影响通过率和 readiness 的问题
- 决策：`ReviewPack` 只收敛当前容量内最有价值的一批任务，不等于活跃任务全集
- 原因：任务过多时必须裁剪，任务过少时也不能让复习入口空掉，系统需要给出稳定、可执行的一包内容
- 决策：先定义 `completion`，再定义 `readiness`
- 原因：`completion` 更适合表达学习过程覆盖到了哪一步，`readiness` 再表达是否接近考试状态
- 决策：`completion` 采用“覆盖完成度 + 初步掌握”的模型
- 原因：如果只看覆盖，太容易出现“学过但没掌握”；如果要求稳定掌握，又会和 `readiness` 过度重叠
- 决策：`readiness` 由 mock 成绩、关键覆盖、高风险薄弱点、关键 review 完成情况与表现稳定性共同卡硬门槛
- 原因：`readiness` 目标应接近真实通过概率，而不是奖励偶然高分
- 决策：`readiness` 目标按 P90 最差情况仍应较大概率通过考试来校准
- 原因：这样更符合“保守但可信”的 ready 判断原则，具体阈值后续参数化
- 决策：mock 错题回流到 review 时应赋予更高权重
- 原因：mock 更接近正式考试环境，暴露出的错误更有代表性
- 决策：如果 mock 暴露的问题此前已有 `MistakeRecord`，则提高后续复习强度
- 原因：这说明问题不是偶发失误，而是持续性薄弱点
- 决策：mock 表现很好时只降低相关易错项优先级，不直接清空 review
- 原因：好的 mock 结果不等于所有薄弱点都已稳定消失
